01. 쿠키 / 스토리지 에디터

tools/storage.ts · panels/CookiesPanel.tsx Tier 1 MCPUI

활성 페이지의 쿠키와 localStorage·sessionStorage를 표 형태로 보면서 인라인 편집할 수 있는 패널입니다. 더블클릭으로 값 수정, 모달에서 Secure·HttpOnly·SameSite·expires까지 변경. AI 에이전트는 같은 동작을 cookie_set으로 호출합니다.

왜 필요한가 쿠키·세션 변조는 거의 모든 인증 우회 챌린지의 첫 단계입니다. DevTools는 클릭 한 번에 값만 바꿀 수 있지만, base64/JWT 인코딩된 값을 디코딩하며 편집하거나, AI에게 "user_lv 쿠키 11로 바꿔줘"라고 자연어로 시키려면 전용 surface가 필요합니다.
MCP 툴
cookie_set(name, value, url?, domain?, path?, secure?, httpOnly?, sameSite?, expires?)
cookie_delete(name, url?, domain?, path?)
cookie_list(urls?)
local_storage_get/set/delete/clear()
session_storage_get/set/delete/clear()
사용 예시
// 사용자 발화
"이 페이지의 user_lv 쿠키를 11로 바꾸고 새로고침해줘"

// 에이전트 동작

02. 헤더 프로파일 주입기

tools/headers.ts Tier 1 MCP

자주 쓰는 HTTP 헤더 조합을 "프리셋"으로 저장해두고, 한 번의 호출로 이후의 모든 매칭 요청에 자동 주입합니다. 내부적으로는 Fetch.requestPaused 인터셉트 룰을 등록하는 방식입니다.

왜 필요한가 "관리자는 내부 IP에서만 접근 가능" 같은 게이트를 우회하려면 X-Forwarded-For: 127.0.0.1을 매번 수동으로 넣어줘야 합니다. 프리셋으로 만들어두면 에이전트가 "트러스트된 IP로 위장해줘"라는 한 줄 지시만으로 활성화할 수 있고, 페이지 이동에도 유지됩니다.
MCP 툴
header_preset_list()
header_preset_save(name, headers, urlPattern?)
header_preset_apply(name)
header_preset_disable(name)
빌트인 프리셋
  • localhost-trust — X-Forwarded-For / X-Real-IP / X-Originating-IP = 127.0.0.1 (내부 IP 화이트리스트 우회)
  • mobile-safari-ua — iPhone Safari 17 UA (모바일 전용 페이지 강제 진입)
  • googlebot — Googlebot 2.1 UA (페이월/봇 화이트리스트)
  • referer-trust — Referer: https://www.google.com/
사용 예시
// 사용자 발화
"이 페이지는 내부 IP만 통과시키는데 우회해줘"

// 에이전트
header_preset_apply({ name: "localhost-trust" })
browser_navigate({ url })
// 이제부터 모든 요청에 X-Forwarded-For: 127.0.0.1 자동 첨부

04. 인트루더 / 블라인드 오라클 ★ 시그니처 기능

tools/intruder.ts Tier 2 MCP

Burp Suite의 Intruder에 해당하는 자동 fuzzer입니다. 베이스 요청의 URL · body · 헤더에 마커 §§를 넣고 페이로드 리스트를 주면, 각 페이로드로 치환된 요청을 브라우저 컨텍스트에서 발사하여 {status, length, durationMs, bodyHash} 표를 돌려줍니다.

왜 필요한가 Blind SQL Injection은 응답 길이·상태 코드·응답 시간의 미세한 차이로 한 글자씩 데이터를 빼내야 합니다. 사람이 수동으로 할 수 없고, Burp Intruder는 유료 (Community는 속도 제한). rever-browser에서 자연어 지시 한 줄로 같은 일을 할 수 있게 만드는 것이 핵심 차별점입니다.
시그니처
intruder_run(requestId, payloads, marker?, concurrency?, diffBaseline?)
진행 흐름
1Repeater에서 베이스 요청을 열고, 변조할 위치에 §§ 마커 삽입
2저장된 requestId로 intruder_run 호출
3결과를 응답 길이 / 응답 시간 / 해시로 정렬하여 outlier 발견
4"이 글자가 정답" 확정 → 다음 자리로 자동 재귀
사용 예시
// 사용자 발화
"old-9의 blind SQLi로 admin 패스워드 첫글자만 알아내줘"

// 에이전트
intruder_run({
  requestId: "req_abc",
  payloads: ["a","b","c","d","e","f","g","h","i","j","k","l","m","n","o","p"],
  marker: "§§"
})

// 결과 (응답 길이 정렬)
| payload | status | length | durationMs |
|---------|--------|--------|------------|
| p       | 200    | 4231   | 110        |  ← outlier!
| a       | 200    | 4180   | 105        |
| b       | 200    | 4180   | 108        |
| ...     |        |        |            |

→ "첫글자는 p" 확정. 두 번째 자리에 § 옮겨서 재귀.
매핑 챌린지
old-2쿠키 blind old-9하드 blind old-13100% blind old-21로그인 blind old-22MD5 reverse old-29필터 blind old-55LEFT() hex old-57time-based + 거의 모든 SQLi 챌린지

시그니처

{
  count: 30,
  wallClockMs: 612,
  statusCounts: { 200: 27, 409: 3 },   ← 정상이면 200=1, 409=29 이어야 함
  timing: { minMs: 102, p50Ms: 215, p95Ms: 480, maxMs: 612 },
  results: [...]
}
// 27번이 통과 → race 취약. 응모 27배 가능.

솔직한 한계 (사용자 검토용)
① header_preset은 기존 intercept-modify 경로를 사용합니다. XHR/fetch는 잘 잡히지만, document 레벨 요청 (페이지 진입 첫 GET)까지 헤더가 주입되는지는 빌드에서 확인이 필요합니다.
② ws_send는 ws_capture_enable 이후 새로 만들어진 소켓만 잡힙니다. 이미 연결된 소켓이 있으면 페이지 reload가 필요합니다.
③ deob_auto의 hex-array / Dean Edwards packer 처리는 regex 기반 휴리스틱입니다. 무거운 일은 webcrack이 하고, 손으로 만든 별난 packer는 수동 한 스텝이 필요할 수 있습니다.
④ burst_send는 페이지의 fetch()를 경유합니다. 진짜 저수준 race 타이밍이 필요하면 Node 측 replay_request가 더 깨끗합니다.
