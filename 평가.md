## 🟢 잘 만든 점

### 1. 컨셉이 명확하고 시장성이 있다 ⭐⭐⭐⭐⭐
"AI agent browser for web API reverse engineering" — 빈 자리 정확.
- 기존: Burp(중간자 프록시), DevTools(수동), Charles(트래픽만) → AI 통합 없음
- rever-browser: **CDP + MCP + AI 에이전트** 한 패키지 = 차별점 명확

### 2. 아키텍처 선택이 적절 ⭐⭐⭐⭐
```
Electron (UI/webview) ─┬─ CDP (브라우저 제어/관찰)
                       ├─ MCP 서버 (AI 도구 노출)
                       └─ ACP (AI 클라이언트 프로토콜)
```
- MCP를 도구 인터페이스로 채택 = 모델 교체 자유로움 (OpenAI, Anthropic, 로컬 모델)
- CDP 직접 = puppeteer/playwright 추상화 안 거치고 raw control
- electron-vite + React 19 + Zustand = 모던하고 가벼움

### 3. 코드 분리가 좋음 ⭐⭐⭐⭐
```
src/main/mcp/tools/  ← 12개 도구 파일 분리
```
한 파일 100~250줄 = 적정선. traffic-store도 단일 책임.

### 4. MCP 도구 카탈로그가 풍부 ⭐⭐⭐⭐
정적 분석 + 동적 분석 + 인증 추출 + 코드 생성 카테고리 골고루.

### 5. 자기 비판이 있다 ⭐⭐⭐⭐⭐
`REVERSING_TOOLS_PLAN.md` 자체가 본인 프로젝트 깐 문서. 검증 에이전트 회고("내 실수 3개")까지 적은 건 드물고 좋은 습관.

## 🟡 아쉬운 점

### 1. README가 없다 ❌
package.json만 있고 진입 문서 없음. 6개월 뒤 본인도 모름. 협업/공유 시 0점.
→ "무엇을 위한 도구인가 / 어떻게 쓰는가 / 데모 GIF" 3줄 필요.

### 2. 테스트가 0개 ❌
tools/ 12개 = 핵심 로직인데 테스트 0. 특히:
- `export_python_client` — Python 코드 생성 정확도
- `auth_dump` — 쿠키 파싱
- `decode` — JWT/base64
단위 테스트 매우 적합. Vitest 셋업 안 되어 있음.

### 3. MCP 서버에 인증 없음 🚨
```ts
server.listen(0, '127.0.0.1', () => resolve())
```
localhost 바인딩만, 토큰/오리진 검증 없음. 같은 머신의 다른 프로세스가 MCP 도구 다 호출 가능.
→ **세션 토큰 + Origin 헤더 검증** 필요.

### 4. MCP 세션당 새 McpServer 인스턴스 생성 ⚠️
```ts
const mcp = buildMcpServer()  // POST마다
await mcp.connect(transport)
```
세션마다 12개 도구 재등록 = 메모리/CPU 낭비.

### 5. traffic-store가 전역 모듈 상태 ⚠️
- 테스트 격리 어려움
- 멀티 윈도우/탭 격리 불가
- 응답 본문 통째로 보관 (MAX_ENTRIES=500이지만 큰 페이지 한 번이면 수백 MB)
→ 응답 본문은 별도 LRU + 사이즈 캡 필요.

### 6. PLAN.md 우선순위 vs 실제 구현 순서 미스매치 의심
PLAN에서 "1순위 = initiator.stack, 2순위 = auth_dump+export_python_client" 라고 본인이 적었는데, 이미 12개 도구 다 만들어져 있음 = 우선순위 무관하게 다 짠 것. 회고 후에도 같은 패턴 반복 가능성.

### 7. 에러 처리가 try/catch 한 덩어리 ⚠️
`tools/*.ts` 다 `try {...} catch (e) { return err(errorMessage(e)) }` 한 줄.
CDP 도메인별 에러 분기 없음 → AI한테 "왜 실패했나" 답을 못 줌.

### 8. `export_python_client`가 너무 순진하다 ⚠️
- JSON 본문이면 `json=` 써야 함 (Content-Type 자동)
- multipart 처리 못 함
- 모든 헤더(User-Agent 등) 그대로 = 서버 거부 가능
→ 현실적으로 안 돌아가는 코드 생성 비율 높을 것.

### 9. 봇 탐지 우회 = 회색지대
`navigator.webdriver` 패치 등 stealth 기능. 합법적 리버싱(본인 사이트, 버그 바운티)은 OK인데 남의 사이트 자동화에 쓰면 ToS 위반.
→ 배포 시 disclaimer / 타겟 도메인 화이트리스트 필요.

## 🔴 구조적 우려

### 1. product의 "Done" 정의가 모호
"AI agent browser" — 어디까지가 v1? 도구 개수가 아니라 demo scenario 3개로 합격선 정의 필요.
예시: "토스 로그인 API 추출", "특정 사이트 검색 API 재현", "JWT 만료 추적".

### 2. AI가 12개 도구를 "잘 쓸지" 불확실
도구 많음 → 컨텍스트 낭비 + 잘못된 선택. 각 도구 description AI 친화적 재작성, 도구 사용 평가셋(eval suite) 필요.

### 3. 사용자와 AI 권한 경계 불명확
AI가 `inject_script` / `Debugger.setScriptSource` 호출 = 임의 코드 실행. 위험도 분류(read-only / mutation / code-exec) 명시 필요.
